### 3. Strategic Analysis (The "So What?")

- **Root Cause:** **Plan quality dominated.** The plan was rigorously defensive: it (a) preserved every parent autotune config (mathematical floor guarantee via the autotune-superset property), (b) provided a written equivalence proof for the routing refactor, and (c) issued an explicit anti-regression checklist. The executor followed it faithfully, producing exactly the prescribed structure. The +1.17% gain is small but precisely in the predicted 0.03–0.08ms savings range — confirming the bottleneck model is accurate.
- **Key Insight:** The lineage has entered a **diminishing-returns plateau** for incremental PyTorch-side micro-optimizations. The GEMM kernels are now genuinely well-tuned (FP8 TC + tl.range pipelining + comprehensive autotune); further large gains will require **structural changes** (fused Triton routing, persistent/worklist GEMM2, TMA via `tl.make_block_ptr`), not refactoring. Notably, `nonzero` removal yielded only ~1.7% — much less than the 2-5% one might naively expect — suggesting routing is no longer the dominant fraction of the 1.52ms budget.
- **Identified Risk:** **Eval-time inflation (8.6x)** is a serious hidden cost — autotune search over 14+13 configs took 591s. If this trend continues with each iteration adding configs, autotune time will eventually dominate and may hit budget/timeout limits. Additionally, the "floor guarantee" strategy creates **monotonic config bloat** — future planners may keep appending without pruning, leading to combinatorial explosion. The numerical error remains in the 30k absolute / 1e11 relative range, still safely within tolerance but a latent cliff if tolerances tighten.

### 4. Actionable Guidance (The "What's Next?")

- **Recommend Exploration (high priority):** **Fused Triton routing kernel.** With PyTorch routing micro-optimizations now exhausted (+1.17% from a clever refactor), the next major lever is consolidating the entire routing pipeline (sigmoid → group top-2 → top-4 groups → masked top-8 → renorm → scatter-sort) into 1-2 Triton kernels with atomic CSR construction. Expected payoff: 3-8% on small-T decode workloads where routing is a larger latency fraction.
- **Recommend Exploration:** **TMA via `tl.make_block_ptr` for weight loads in GEMM1.** GEMM1 (K=7168) is the dominant kernel cost (~55% of 1.52ms). Blackwell TMA can deliver 1.2-1.5x bandwidth on weight tiles — potential 5-10% overall speedup. This is the highest-value untapped Triton 3.6.0 / B200 feature.
- **Recommend Stripping:** **Prune autotune configs.** Before adding more configs, future planners should examine the autotune cache to identify which configs are NEVER selected by the autotuner across all 19 workloads, and remove them. This will cut eval-time inflation without risking the performance floor. A target of ≤8 configs per kernel is reasonable.
- **Recommend Fusion:** **Fuse the `output = zeros(); index_add_(); .to(bf16)` epilogue** into the GEMM2 kernel via `tl.atomic_add` directly into a bf16 output buffer. Eliminates a full 7168×T scratch + a separate launch. Risk: bf16 atomic precision; mitigation: keep fp32 atomic accumulator + post-cast.
- **Warn:** Do NOT continue the "verbatim copy + tiny tweak" pattern indefinitely. Each iteration here yielded +0.018 score; at this rate we need 25+ iterations to gain another 0.5x. The strategy that broke through at iteration 31 (FP8 TC fusion, +0.31 absolute) was structural, not cosmetic. Planners must be willing to accept higher variance for the chance at structural wins.
- **Warn:** Do NOT regress the GEMM2 FP8 TC triad (`tl.max(tl.abs(c))` scalar / `tl.float8e4nv` cast / `tl.dot(c_fp8, w2_fp8)`) or the `argsort(..., stable=True)` in the routing refactor — the latter is essential for tie-breaking equivalence with the original `nonzero+argsort` ordering, and silent unstable sorting could cause weight/token mis-pairing under specific routing patterns.
