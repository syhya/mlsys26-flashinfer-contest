2. **Plan 1 (Triton fused routing) is too risky.** DeepSeek no-aux routing (group top-2 sum → top-4 groups → masked top-8 → renorm by raw sigmoid) has fragile tie-breaking and mask semantics. A single wrong ordering breaks correctness silently.

## Best Plan

### Phase 1: Mathematical Definition

**Step 1.1: Full pipeline math (unchanged from 8e3c776e):**
```
q[t,e]           = sigmoid(logits[t,e])                               # raw score (used for weights)
rank[t,e]        = q[t,e] + bias[e]                                   # biased score (used for ranking)
group_score[t,g] = sum of top-2 of rank[t, g*32:(g+1)*32]
top_groups[t]    = argmax_{4 groups} group_score[t]
mask[t,e]        = 1 iff e ∈ selected groups
topk8[t]         = top-8 of (rank where mask=1 else -inf)
w[t,e]           = q[t,e] / sum_{e' ∈ topk8[t]} q[t,e']  *  scaling    (for e ∈ topk8[t])
```

**Step 1.2: GEMM2 FP8 TC path (unchanged from 8e3c776e):**
```
c       = x1 * x2 * sigmoid(x2)                         # SwiGLU fp32 tile
cmax    = tl.max(tl.abs(c))                              # SCALAR, no axis
cs      = max(cmax / 448.0, 1e-12)                       # SCALAR
c_fp8   = (c / cs).to(tl.float8e4nv)                     # saturating cast
acc    += tl.dot(c_fp8, w2_fp8, out_dtype=fp32) * cs * w2_scale
```

**Variables (X):** autotune configs (tile sizes, num_warps, num_stages); routing op sequence.
**Constraints (C):** correctness ≡ bit-close to PyTorch ref within atol=rtol=1e-2; BLOCK_N=128 in all GEMM2 configs; int64 pointer math; `torch.zeros` for G1/O/output; scalar `tl.max` in GEMM2.
**Objective f(X):** minimize kernel latency (maximize speedup), floor = `8e3c776e`.

**Step 2.3: Refactor `_route_tokens` to reduce launch count (MATHEMATICALLY IDENTICAL):**

Replace the sequence `torch.nonzero(expert_mask, as_tuple=True) -> argsort(local_expert_idx) -> index with flat_token_idx[order]` with the equivalent but single-pass approach below. Reasoning: `nonzero` triggers a device sync AND produces output whose size is not statically known; replacing with a materialized dense index avoids the sync in the critical path.

Original block (in `8e3c776e`):
```python
expert_mask = (topk_idx >= local_start) & (topk_idx < local_start + e_local)
flat_token_idx, flat_k_idx = torch.nonzero(expert_mask, as_tuple=True)
if flat_token_idx.numel() == 0:
    return None
local_expert_idx = topk_idx[flat_token_idx, flat_k_idx] - local_start
order = torch.argsort(local_expert_idx)
sorted_tokens  = flat_token_idx[order].to(torch.int32)
sorted_experts = local_expert_idx[order].to(torch.int32)
sorted_weights = weights[flat_token_idx, topk_idx[flat_token_idx, flat_k_idx]][order].to(torch.float32)
```

Replacement (same math, fewer syncs, no `nonzero`):
```python
TOP_K = 8
# Build dense [T*TOP_K] triples in one vectorized pass
flat_expert = topk_idx.reshape(-1)                                 # [T*8] int64
flat_token  = torch.arange(T, device=device).repeat_interleave(TOP_K)  # [T*8]
# Indexing weights with (token, expert) pairs via gather:
flat_weight = weights.gather(1, topk_idx).reshape(-1)              # [T*8] f32

# Mask to local experts; replace masked-out expert with sentinel large value so sort pushes them to end.
local_start = int(local_expert_offset)
shifted = flat_expert - local_start                                # [T*8]
in_range = (shifted >= 0) & (shifted < e_local)                    # [T*8] bool
# Use a sentinel = e_local (sorted after all valid indices)
sort_key = torch.where(in_range, shifted, torch.full_like(shifted, e_local))

# Single sort produces both grouping + ordering
order = torch.argsort(sort_key, stable=True)
sort_key_sorted = sort_key[order]

# Count valid entries (one scalar CPU sync — fused with the existing max_count sync later)
num_valid = int(in_range.sum().item())    # ALREADY have one sync in run() for max_count; combine into one .item() path
if num_valid == 0:
    return None

valid_order = order[:num_valid]
sorted_tokens  = flat_token[valid_order].to(torch.int32)
sorted_experts = shifted[valid_order].to(torch.int32)
sorted_weights = flat_weight[valid_order].to(torch.float32)

expert_counts  = torch.bincount(sorted_experts, minlength=e_local)
expert_offsets = torch.zeros(e_local + 1, dtype=torch.int32, device=device)
expert_offsets[1:] = torch.cumsum(expert_counts, dim=0)
```

Mathematical equivalence proof:
- `nonzero+argsort` produced pairs `(t,k)` where `expert_mask[t,k]=True`, sorted by `(local_expert_idx, row_major_tie_breaker)`.
- The refactor produces the same `(t, expert_idx - local_start)` pairs: iterating `(t,k)` in row-major order equals `arange(T).repeat_interleave(8)` for tokens and `reshape(-1)` for experts. Sentinel sentinel-sort pushes masked-out pairs to the tail; we slice them off with `[:num_valid]`. `argsort(stable=True)` preserves row-major tie-breaking → identical ordering.
- `weights.gather(1, topk_idx).reshape(-1)` equals `weights[flat_token_idx, topk_idx[flat_token_idx, flat_k_idx]]` when reading in row-major order.
