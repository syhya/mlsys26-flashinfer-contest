## Phase 1: Mathematical Definition

### 1.1 Full computation model
The child must still compute exactly the same MoE function:

1. **Routing**
   - `q[t,e] = sigmoid(routing_logits[t,e])`
   - `rank[t,e] = q[t,e] + routing_bias[e]`
   - Partition 256 experts into `N_GROUP = 8` groups of size 32.
   - For each token `t` and group `g`, define:
     - `group_score[t,g] = sum(top2(rank[t, group g]))`
   - Select top-4 groups per token.
   - Mask all experts outside those groups.
   - Select global top-8 experts from the masked `rank`.
   - Define routing weights using **raw sigmoid scores** `q`, renormalized only over the selected top-8 experts, then multiplied by `routed_scaling_factor`.

2. **GEMM1**
   - For each routed token/expert pair `j` with original token index `tok[j]` and expert `e[j]`:
   - `G1[j,n] = Σ_k hidden_states[tok[j],k] * gemm1_weights[e[j],n,k] * hidden_states_scale[k//128, tok[j]] * gemm1_weights_scale[e[j], n//128, k//128]`
   - Accumulation must remain FP32.

3. **SwiGLU**
   - Split `G1` into `x1 = G1[:, :I]`, `x2 = G1[:, I:]` with `I = 2048`
   - `C[:,i] = x1[:,i] * x2[:,i] * sigmoid(x2[:,i])`

4. **GEMM2**
   - `O[j,h] = Σ_i C[j,i] * gemm2_weights[e[j], h, i] * gemm2_weights_scale[e[j], h//128, i//128]`
   - Keep the current-best FP8-TC GEMM2 math from `6da80c61` unchanged.

5. **Scatter-add**
   - `output[tok[j], h] += routing_weight[j] * O[j,h]`
   - Final output dtype must be `torch.bfloat16`.

### 1.2 Variables
Define optimization variables:
- `R`: routing implementation.
- `K1`: GEMM1 implementation.
- `K2`: GEMM2 implementation.
- `E`: epilogue/output accumulation path.

For this child:

- `K1 = exact 6da80c61 GEMM1 path`

- Expert/token dispatch fully GPU-vectorized

The replacement must follow this exact mathematical structure:

1. Compute `q`, `rank_scores`, `group_scores`, `top_groups`, `expert_allowed`, `pruned`, `topk_idx`, `weights` exactly as usual.
2. Replace the local expert extraction block based on:
   - `torch.nonzero(expert_mask, as_tuple=True)`
   - `argsort(local_expert_idx)`

with the dense flattened method:

```python
flat_expert = topk_idx.reshape(-1)
flat_token = torch.arange(T, device=device, dtype=torch.int64).repeat_interleave(TOP_K)
flat_weight = weights.gather(1, topk_idx).reshape(-1)

shifted = flat_expert - local_start
in_range = (shifted >= 0) & (shifted < e_local)
sentinel = torch.full_like(shifted, e_local)
sort_key = torch.where(in_range, shifted, sentinel)
order = torch.argsort(sort_key, stable=True)

n_valid = int(in_range.sum().item())
if n_valid == 0:
    return None

order = order[:n_valid]
sorted_tokens = flat_token[order].to(torch.int32)
sorted_experts = sort_key[order].to(torch.int32)
sorted_weights = flat_weight[order].to(torch.float32)

expert_counts = torch.bincount(sorted_experts, minlength=e_local)
expert_offsets = torch.zeros(e_local + 1, dtype=torch.int32, device=device)
expert_offsets[1:] = torch.cumsum(expert_counts, dim=0)
```

### 3.2 Exact routing equivalence rule
The dense sentinel-sort method is valid because it reproduces the same local routed tuple set as the original sparse extraction:
- `topk_idx.reshape(-1)` enumerates `(token, k)` in row-major order.

- `sort_key = shifted or sentinel` groups local experts first and non-local experts last.
