Secondary insight: **TMA gains are real but smaller than theoretical** on this workload — likely because GEMM1 is already well-pipelined with `num_stages=4-5`, leaving less bandwidth to unlock.

Unexpected, since TMA loads should be numerically identical to raw-pointer loads. Likely explanation: autotune now selects slightly different tile configs for the TMA kernel, changing the order of fp32 accumulation.
