## Phase 1: Mathematical Definition

### 1.1 Variables
Let:
- `T` = number of tokens.
- `E_global = 256` total experts.
- `E_local = 32` local experts in this shard.
- `H = 7168` hidden size.
- `I = 2048` intermediate size.
- `B = 128` block scale size.
- `q[t,e] = sigmoid(routing_logits[t,e])`.
- `r[t,e] = q[t,e] + routing_bias[e]` be the ranking score only.

Define group structure:
- `N_GROUP = 8`
- `GROUP_SIZE = 32`
- `TOPK_GROUP = 4`
- `TOPK_EXPERT = 8`

For each token `t`:
1. Reshape `r[t,:]` into 8 groups of 32 experts.
2. For each group `g`, define
   `group_score[t,g] = sum(top2(r[t, group g]))`.
3. Select top-4 groups.
4. Among experts only in those groups, select top-8 experts globally by `r[t,e]`.
5. Define binary routing mask `M[t,e] in {0,1}` for those selected experts.
6. Define normalized routed weights
   `w[t,e] = routed_scaling_factor * q[t,e] * M[t,e] / (sum_j q[t,j] * M[t,j] + 1e-20)`.

Local dispatch variables:
- Flatten only local routed pairs `(t, e_local)`.
- Sort by local expert id.
- Let `Tk_total` be the total number of local routed token-expert pairs.
- Let:
  - `sorted_tokens[j]` = token id for routed row `j`
  - `sorted_experts[j]` = local expert id for routed row `j`
  - `sorted_weights[j]` = routing weight for routed row `j`
- Let CSR offsets satisfy
  `expert_offsets[e+1] - expert_offsets[e] = count of rows for expert e`.

For each routed row `j`, with token `t = sorted_tokens[j]` and local expert `e = sorted_experts[j]`:

### 1.2 GEMM1 exact equation
Input hidden states are FP8 with per-token, per-K-block scales:
- `hidden_states[t,k]` in FP8.
- `hidden_states_scale[kb, t]`, where `kb = k // 128`.

First projection weights are FP8 with per-output-block and per-K-block scales:
- `gemm1_weights[e,n,k]` in FP8, for `n in [0, 4096)`.
- `gemm1_weights_scale[e, nb, kb]`, where `nb = n // 128`.

Then:
`G1[j,n] = sum_{k=0}^{H-1} hidden_states[t,k] * gemm1_weights[e,n,k] * hidden_states_scale[k//128, t] * gemm1_weights_scale[e, n//128, k//128]`
for `n in [0, 2I)`.

### 1.3 SwiGLU exact equation
Split:
- `x1[j,i] = G1[j,i]`
- `x2[j,i] = G1[j,I+i]`

Then:
`C[j,i] = silu(x2[j,i]) * x1[j,i]`
for `i in [0, I)`.

### 1.4 GEMM2 exact equation
Second projection weights:
- `gemm2_weights[e,h,i]` in FP8
- `gemm2_weights_scale[e, hb, ib]`, where `hb = h // 128`, `ib = i // 128`

Then:
`O[j,h] = sorted_weights[j] * sum_{i=0}^{I-1} C[j,i] * gemm2_weights[e,h,i] * gemm2_weights_scale[e, h//128, i//128]`
for `h in [0, H)`.

### 1.5 Final accumulation equation
Final output:
`Y[t,h] = sum_{j : sorted_tokens[j] = t} O[j,h]`

Return:
`Y.to(torch.bfloat16)`.

### 1.6 Hard constraints
The child solution must satisfy all of the following:
1. Exact DeepSeek routing semantics.
2. Bias used only for ranking, never for final weight normalization.
3. FP8 block-scale interpretation is multiplicative over 128-sized blocks.
4. Hidden-state scales indexed by original global token id, not grouped row id.
5. All GEMM accumulation in float32.
6. Final output shape `[T, 7168]`, dtype `torch.bfloat16`.
7. Works for arbitrary `T`.
8. No Python loops over routed rows that trigger repeated CPU-GPU synchronization.

### 1.7 Objective function
Primary objective:
- Remove the persistent large numerical discrepancy.

Secondary objective:
- Retain substantial speedup over the PyTorch reference.

Optimization target:
- Minimize total runtime subject to exact forward semantics and evaluator correctness tolerance.

## Phase 2: The Optimization Loop

This is a deterministic staged optimization process. There is no randomness, so no multi-start loop is required.

### Step 2.1: Build exact routing and CSR dispatch in vectorized PyTorch
Implement a helper such as `_route_tokens(...)` using only PyTorch tensor ops.

Required operations:
- `torch.sigmoid`
- `torch.topk(..., sorted=True)`
- `torch.where`
- `torch.nonzero(as_tuple=True)`
- `torch.argsort`
- `torch.bincount`
- `torch.cumsum`
- `scatter_`

Exact procedure:
1. Compute `q = torch.sigmoid(routing_logits.float())`.
2. Compute ranking scores `rank_scores = q + routing_bias.float().view(1, E_global)`.
3. Reshape into `[T, 8, 32]`.
4. Compute top-2 within each group, sum them into `group_scores`.
5. Select top-4 groups.
6. Build a boolean expert-allowed mask.
7. Replace disallowed experts with `-inf`.
8. Select top-8 experts globally from the allowed set.
9. Construct the final normalized routing weights using `q`, not `rank_scores`.
10. Keep only local experts using `local_expert_offset`.
11. Flatten local routed pairs and sort by local expert id.
12. Build `expert_counts` via `torch.bincount`.
13. Build `expert_offsets` via `torch.cumsum`.

This routing stage is the structural backbone. It must remain mathematically explicit and untouched by later speed optimizations.

### Step 2.2: Use Triton only for grouped GEMM1
Retain the fast grouped GEMM1 design because it is likely not the root cause of the persistent 8192 error, and it is where FP8 tensor cores are most beneficial.

Kernel strategy:
- Use `@triton.autotune` with a small fixed config set.
- Use `BLOCK_K = 128` so one K tile aligns exactly with one hidden-state scale block.
- Keep FP8 operands in native dtype for `tl.dot`.
- Apply `a_scale * w_scale` after each tile dot accumulation in FP32.

Recommended config set:
- `(BLOCK_M=32, BLOCK_N=128, num_warps=4, num_stages=4)`
- `(BLOCK_M=64, BLOCK_N=128, num_warps=4, num_stages=3)`
- `(BLOCK_M=64, BLOCK_N=64, num_warps=4, num_stages=4)`
- `(BLOCK_M=128, BLOCK_N=128, num_warps=8, num_stages=3)`

Launch model:
- Grid dimensions:
  - `grid_x = ceil_div(4096, BLOCK_N)`
  - `grid_y = ceil_div(max_count, BLOCK_M)`
  - `grid_z = E_local`
- Inside kernel:
  - recover `Tk = expert_offsets[e+1] - expert_offsets[e]`
  - return early if `m_idx * BLOCK_M >= Tk`

Critical kernel requirements:
1. Convert all large offset arithmetic to `tl.int64`.
2. `token_ids` must come from `sorted_tokens`.
3. Hidden-state scale lookup must use the original token id.
4. Every partial N tile must be masked.
5. Every final store must use `m_mask & n_mask`.

Output:
- `G1` shape `[Tk_total, 4096]`, dtype `float32`.

### Step 2.3: Replace Triton GEMM2 with exact per-expert PyTorch GEMM2
This is the key algorithmic change.

Instead of continuing with a potentially flawed custom grouped Triton GEMM2, do the following:
1. For each expert `e`, extract row range `[s:t) = [expert_offsets[e], expert_offsets[e+1])`.
2. If `s == t`, continue.
3. Slice `G1_e = G1[s:t]`.
4. Split `x1 = G1_e[:, :I]`, `x2 = G1_e[:, I:]`.
5. Compute `C_e = torch.nn.functional.silu(x2) * x1` in float32.
6. Dequantize only `gemm2_weights[e]` exactly into float32 using block scales.
7. Compute `O_e = C_e @ W2_e.T` in float32.
8. Multiply `O_e *= sorted_weights[s:t].unsqueeze(1)`.
9. Scatter-add to final output with `index_add_`.

This turns GEMM2 into a deterministic mathematical oracle for the second projection while still avoiding full-model global dequantization.

### Step 2.4: Dequantize W2 exactly and locally
Implement a helper such as `_dequant_w2_expert(e)`.

Exact dequantization rule:
- `gemm2_weights[e]` has shape `[7168, 2048]`.
- `gemm2_weights_scale[e]` has shape `[56, 16]`.
- Expand scales with:
  - `repeat_interleave(128, dim=0)` over H blocks
  - `repeat_interleave(128, dim=1)` over I blocks
- Slice to exact shape and multiply with `gemm2_weights[e].float()`.

Example implementation direction:
- `scale = gemm2_weights_scale[e].float()`
- `scale_exp = scale.repeat_interleave(128, 0).repeat_interleave(128, 1)`
- `W2_e = gemm2_weights[e].float() * scale_exp`

This explicit construction eliminates ambiguity in orientation and block indexing.

### Step 2.5: Optional exact GEMM1 micro-check before trusting fast path
To mathematically localize errors, add a tiny deterministic validation check in the fast path:
1. Pick the first active expert `e0`.
2. Pick a very small number of its routed rows, e.g. `m = min(2, Tk_e0)`.
3. Compute those `m` GEMM1 rows both ways:
   - via the produced Triton `G1`
   - via exact PyTorch dequantized GEMM1 for that expert and those tokens
4. Compare max error.
5. If mismatch exceeds a strict threshold, abort fast path and use full fallback.

This is not random. It is a deterministic semantic guard.

### Step 2.6: Final accumulation with deterministic segmented reduction
Use only:
- `output = torch.zeros((T, H), dtype=torch.float32, device=device)`
- `output.index_add_(0, token_ids, contributions)`

Do not attempt a custom segmented reduction kernel in this child. The goal is robustness.

## Phase 3: The "Safety Valve" (CRITICAL)

This phase is mandatory. It ensures the child always returns a valid output even if the accelerated path is semantically wrong or unsupported.

### Step 3.1: Structural validation before any Triton launch
Before running GEMM1, validate:
1. `sorted_tokens.numel() == sorted_experts.numel() == sorted_weights.numel()`
2. `expert_offsets.shape[0] == E_local + 1`
3. `expert_offsets[0] == 0`
4. `expert_offsets[-1] == Tk_total`
5. `expert_offsets` is monotonic nondecreasing
6. `expert_counts == expert_offsets[1:] - expert_offsets[:-1]`

If any check fails:
- skip fast path entirely,
- run exact fallback.

### Step 3.2: Kernel masking and pointer safety rules
Inside Triton GEMM1, enforce all of the following:
- `m_mask = m_offs < Tk`
- `n_mask = n_offs < 4096`
- all expert pointer arithmetic in `tl.int64`
- output stores masked with `m_mask[:, None] & n_mask[None, :]`

This prevents partial-tile corruption and overflow.

### Step 3.3: Deterministic semantic micro-check
After GEMM1 completes, but before committing to final output:
- run the exact small GEMM1 validation described in Phase 2.5.
- If the check fails, discard fast-path result and run fallback.

This is the core correctness refinement mechanism. It is analogous to a projection/repair step in optimization: the accelerated candidate is accepted only if it lies in the validity region defined by the exact equations.

### Step 3.4: Exact fallback implementation
Implement `_fallback_run(...)` that computes the whole local MoE path exactly in PyTorch.

Fallback steps:
1. For each local expert `e`:
   - gather routed token ids
   - exact-dequant hidden-state rows for those tokens
   - exact-dequant `gemm1_weights[e]`
   - compute `G1 = A @ W1.T`
   - compute `C = silu(G1[:, I:]) * G1[:, :I]`
   - exact-dequant `gemm2_weights[e]`
   - compute `O = C @ W2.T`
   - multiply by routing weights
   - `index_add_` into output
2. Return `output.to(torch.bfloat16)`.

This fallback is guaranteed-valid because it directly implements the mathematical equations.

### Step 3.5: Conservative fast-path acceptance rule
The child should prefer correctness over speed.

Therefore:
- If Triton compilation fails, fallback.
- If GEMM1 micro-check fails, fallback.
- If any shape/stride inconsistency appears, fallback.
- If `Tk_total == 0`, return zeros immediately.

This ensures the child never returns a malformed tensor or silently wrong output.

## Phase 4: Implementation Details

### 4.1 Allowed libraries
Use only:
- `torch`
- `triton`
- `triton.language as tl`

No obscure packages. No external solver dependency is required.

### 4.2 Required helper functions
The executor should implement these helpers explicitly:
1. `_route_tokens(...)`
2. `_dequant_hidden_rows(hidden_states, hidden_states_scale, token_ids)`
3. `_dequant_w1_expert(gemm1_weights_e, gemm1_weights_scale_e)`
4. `_dequant_w2_expert(gemm2_weights_e, gemm2_weights_scale_e)`
5. `_validate_dispatch(...)`
6. `_gemm1_microcheck(...)`
7. `_fallback_run(...)`

### 4.3 Exact helper implementation details
#### `_dequant_hidden_rows(...)`
Input:
- `hidden_states[token_ids]` shape `[m, 7168]`
- `hidden_states_scale[:, token_ids]` shape `[56, m]`

Implementation:
1. transpose scales to `[m, 56]`
2. `repeat_interleave(128, dim=1)` to `[m, 7168]`
3. multiply by `hidden_states[token_ids].float()`

#### `_dequant_w1_expert(...)`
Input:
- weight shape `[4096, 7168]`
- scale shape `[32, 56]`

Implementation:
1. expand scale across rows and columns with `repeat_interleave(128, dim=0/1)`
2. multiply by FP8 weight cast to float32

#### `_dequant_w2_expert(...)`
Input:
- weight shape `[7168, 2048]`
- scale shape `[56, 16]`

Implementation:
1. expand by 128 in both dims
2. multiply by FP8 weight cast to float32

### 4.4 Triton GEMM1 kernel specifics
Use a single grouped kernel with parameters like:
- `A_ptr`, `A_scale_ptr`
- `W_ptr`, `W_scale_ptr`
- `Out_ptr`
- `sorted_tokens_ptr`
- `expert_offsets_ptr`

Key indexing rules:
- `token_ids = load(sorted_tokens_ptr + start_idx + m_offs)`
- A pointers use original token ids
- scale lookup uses `hidden_states_scale[k_block, token_id]`
- W scale lookup uses `gemm1_weights_scale[e, n_block, k_block]`

Use `tl.dot(a, w, out_dtype=tl.float32)` and then multiply by scales.

### 4.5 `run()` function structure
The executor should implement `run()` in this order:
1. Read `T`, `H`, `I`, `E_local`.
2. Compute routing and dispatch with `_route_tokens`.
3. If no local routed rows, return zero bf16 output.
4. Validate dispatch with `_validate_dispatch`.
5. If validation fails, run `_fallback_run`.
6. Try fast path:
   - allocate `G1`
   - launch Triton grouped GEMM1
   - run `_gemm1_microcheck`
   - if micro-check fails, fallback
   - per expert, do exact PyTorch SwiGLU + dequantized GEMM2 + weighted accumulation
7. On any exception, fallback.
8. Return final output as bf16.

### 4.6 Performance notes for the executor
To preserve speed despite exact GEMM2:
1. Process experts only if `expert_counts[e] > 0`.
2. Reuse dequantized W2 per expert only within that expert block.
3. Avoid dequantizing any unused expert.
4. Keep output accumulation in a single preallocated float32 tensor.
5. Keep all tensors on GPU.

### 4.7 Why this is still performance-competitive
Although exact PyTorch GEMM2 is slower than the current grouped Triton GEMM2, this plan still has a credible path to a better evaluator score because:
- GEMM1 remains accelerated and is large.
- Routing/dispatch stays vectorized on GPU.
- Dequantization is only for routed local experts, not entire global model tensors.
- The dominant objective is to cross the correctness-speed boundary to score >= 1.0, which the current fast-but-wrong lineage has failed to do.
