# Key Optimization Directions

## 1. Leverage FP8 Tensor Cores (Critical)
B200 FP8 tensor cores provide ~4x throughput over FP32. When writing custom Triton GEMM kernels, keep operands in FP8 for `tl.dot` and apply block-scales **after** dot accumulation, rather than casting to FP32 before the dot. When `BK=128` and `BN=128`, each tile aligns to exactly one scale block, so the post-multiply is a cheap scalar operation.

## 2. Fuse FP8 Dequant into GEMM
The reference materialises full fp32 weight tensors upfront -- extremely memory-intensive. Instead, load fp8 tiles and apply block scales during accumulation. This avoids a `[32, 4096, 7168]` f32 scratch buffer (~14 GB) and redundant HBM reads.

## 3. Indirect A-Scale Lookup
`hidden_states_scale` has shape `[H//128, T]` (feature-block x global-token). After gathering token rows, the GEMM kernel must look up scales via the **original global token index**, not the local expert-batch index.

## 4. Expert-Parallel Dispatch
The serial `for le in range(32)` loop serialises expert computation on GPU. A grouped-GEMM approach that uses the expert index as a grid dimension eliminates per-expert launch overhead and maximises SM utilisation. Build dispatch structures (CSR offsets) entirely on GPU using vectorized ops like `torch.bincount` + `cumsum` to avoid CPU-GPU synchronization.

## 5. SwiGLU + FP16 Quantization for GEMM2 Tensor Core Utilization (Critical)
Compute SwiGLU in a standalone kernel and dynamically quantize its output to FP16 (per-row scaling). GEMM2 then runs as FP16×FP16 on tensor cores with a scale correction in the epilogue. B200 FP16 TC throughput is 2x over TF32, making this the highest-impact optimization for GEMM2.

## 6. T-Adaptive Dispatch Strategy
Performance characteristics differ drastically by T. Use programmatic dispatch: small T (decode) benefits from K-splitting to increase parallelism; medium T benefits from moderate tile sizes; large T (prefill) benefits from large tiles and higher occupancy. Select tile sizes, K-split factors, and even kernel variants based on T at launch time.

## 7. Worklist-Based Dispatch (Eliminate Empty CTAs)
Using expert as a grid dimension causes many CTAs to early-exit for experts with few tokens. For GEMM2 especially, build a compact worklist of (expert_id, token_start) pairs covering only active tiles, then launch exactly that many CTAs. This eliminates idle CTA overhead in typical skewed routing distributions.

## 8. Fused Routing Kernel
Multiple PyTorch ops for routing (sigmoid, topk, argsort, bincount, cumsum) incur many kernel launches and implicit syncs. Fuse the entire routing + sort pipeline into one or two Triton kernels using atomic operations and spin-barriers. This is especially impactful for small T where launch overhead dominates.

## 9. Triton Performance Features (Triton 3.6.0)
- **Software pipelining**: Control pipeline depth via `num_stages` in `@triton.Config(...)`. Use standard `for k in range(0, K, BLOCK_K):` loops inside kernels.
- **`tl.make_block_ptr`**: Use block pointers for weight loads to enable hardware TMA (async bulk copy) on Blackwell, providing higher bandwidth than regular pointer arithmetic loads
- **Autotuning**: Use `@triton.autotune` with multiple `triton.Config` entries to find optimal tile sizes, `num_warps`, and `num_stages` per problem shape

## 10. Split-K GEMM for Small Batches
When `Tk` is small (decode regime), a standard GEMM tiles poorly. Split-K approaches with atomic accumulation can keep SMs busy across the K=7168 reduction axis. Use higher K_SPLITS (4-8) for very small T (1-4 tokens) and reduce as T grows.

# Implementation Pitfalls to Avoid
- **int32 pointer overflow**: Expert weight tensors `[32, 4096, 7168]` have offsets exceeding int32 range. Use `int64` for pointer arithmetic in Triton kernels (`.to(tl.int64)`).
- **CPU-GPU sync in dispatch**: Avoid Python loops over GPU tensors for building expert offsets. Each element access triggers a sync. Use `torch.bincount`, `torch.cumsum`, `torch.cummax` instead. Ideally allow at most ONE CPU-GPU sync per forward pass (to read max_tokens for grid sizing), and use async stream copy for that.
- **Dimension constexpr**: Declare known dimension sizes (H=7168, I=2048) as `tl.constexpr` to enable compiler optimizations.
