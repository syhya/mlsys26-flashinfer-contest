Outline 3: Two-tier exact/optimized MoE with deterministic oracle cross-check at stage boundaries

Mathematical Formulation
- Variables (X):
  - R = exact routing outputs.
  - D = exact CSR dispatch.
  - Fast-path intermediate tensors F1_e, F2_e for each expert/chunk.
  - Oracle tensors O1_e, O2_e generated by exact blockwise dequantization.
  - Final output Y.
- Constraints (C):
  1. R and D are exact and identical for both fast and oracle paths.
  2. For every expert batch, fast GEMM stage outputs must satisfy ||F - O|| <= epsilon_stage before enabling that fast stage globally.
  3. Final output uses only validated stages; otherwise oracle stage replaces invalid stage.
  4. All final accumulations are fp32 and output is bf16.
- Objective f(X): maximize speed while maintaining stagewise correctness invariants.

Solver Strategy
- Deterministic staged solver:
  1. Solve routing and dispatch exactly.
  2. Execute candidate optimized stage (grouped Triton or native backend) on a small deterministic calibration subset of dispatched rows/chunks.
  3. Compare against exact oracle stage using max-abs / relative threshold.
  4. Lock in validated fast stage; reject invalid stage and replace with oracle implementation.
  5. Run full workload with chosen stage composition.
- This is a validation-gated algorithm selection procedure, not randomness.

Validity Mechanism
- Hard safety gate at stage boundaries ensures no unvalidated optimized stage is used.
- Oracle path is exact blockwise dequant+fp32 mm, chunked to avoid memory blowups.
- Final fallback is fully exact per-expert chunked execution, which guarantees valid output.

Why this solution?
- Best robustness against the current lineage’s unresolved structural bug because it localizes error to a stage instead of guessing.
- Advantages: decouples structure finding from parameter execution, prevents silent wrong fast paths, preserves opportunity for high speed if one stage validates.
- Disadvantages: more code and some extra overhead for a tiny calibration pass, but still polynomial and practical.
