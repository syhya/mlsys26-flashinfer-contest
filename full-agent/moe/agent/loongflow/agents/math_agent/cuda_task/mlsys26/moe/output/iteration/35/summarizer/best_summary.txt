### 3. Strategic Analysis (The "So What?")

- **Root Cause:** **Plan-level design flaw**, not execution error. The executor faithfully implemented what the plan specified. The plan's correctness argument had a critical hole: it claimed "no data race because each (token, expert) pair appears at most once." While true across the full launch, `tl.atomic_add` into `output[T,H]` from many CTAs produces non-deterministic floating-point summation order — but that alone should stay within 1e-2 tolerance. The more likely culprit is a **duplicate-accumulation bug**: the plan claims `GEMM2 already multiplies routing_w before the final store` but in the parent code, the stored `O` was then consumed by `index_add_` (single accumulation). With the new fused path, the routing_w multiplication remains but the atomic target may be receiving contributions from multiple `pid_n` tiles for the same token (correct), OR the plan may have missed that the `pid_e`-grid launches CTAs for experts with `num_tokens==0` that still load garbage via masked=0, which normally no-ops but with atomic_add into a shared global buffer can interact badly. Alternatively, the plan's assertion that `output = torch.zeros` is sufficient for atomic_add may have failed if `torch.zeros` allocation was not complete before kernel launch on this stream (unlikely, but plausible under async conditions). The plan also ignored the possibility that previous correctness margins were already thin (max_abs_err=32,768 for parent), and FP summation reordering from atomics pushed error past the 1e-2 bound.
- **Key Insight:** The single most important lesson: **mathematical "floor guarantees" provided by an autotune superset DO NOT apply when the kernel body is also modified**. The plan repeatedly invoked "worst case = parent score" as a safety argument — but by changing the store semantics (atomic_add), the floor collapsed to zero the moment validation failed. The `USE_FUSED_ATOMIC_EPILOGUE = True` default with no automated fallback trigger (runtime correctness check) meant the claimed safety valve was inert. **Runtime-defaulted safety toggles without automated fallback are theater, not safety.**
- **Identified Risk:** Two serious systemic risks exposed:
  1. **Planner overconfidence in hand-waved correctness proofs.** The atomic_add commutativity argument was asserted, not tested. Any plan that reorders floating-point summation must be validated against the existing 32,768 max_abs_err ceiling — which was already close to the tolerance edge.
  2. **Lineage-frontier precision debt**: The numerical headroom has been shrinking across generations (8,192 → 32,768 max_abs_err). Any change touching accumulation order (atomic, split-K, reduction refactor) is now high-risk until precision is reclaimed.

This reduces max_abs_err by an expected 2-4x, restoring precision headroom and making future reordering optimizations viable.

- **Warn — CRITICAL:** Future plans that change accumulation semantics (atomic, split-K, reorder) MUST include an **automated runtime correctness fallback**, not a hand-coded boolean flag defaulted to the risky path. Pattern: run both paths on a small probe tensor on first call, compare, fall back if delta exceeds tolerance. The `USE_FUSED_ATOMIC_EPILOGUE = True` toggle in this solution was false safety — it requires a human to notice failure and flip it, which defeats the purpose.

- **Warn:** Do NOT re-attempt atomic_add GEMM2 epilogue until (a) per-row FP8 quant is deployed to reclaim precision headroom, AND (b) the plan includes an explicit proof that summation ordering cannot exceed the measured `max_abs_err` ceiling for the noisiest workload.

- **Warn:** The plan explicitly claimed "mathematical floor guarantee ≥15.52x" — this was **false advertising** because it conflated autotune-superset floor (which is real for latency) with correctness floor (which is non-existent for kernel-body changes). Future planners should be penalized for applying superset arguments to non-superset changes.
