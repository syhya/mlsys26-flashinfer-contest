The child's GEMM1 modification was intended to improve numerical stability by avoiding FP8 tensor-core accumulation, but it actually degraded both error metrics and latency. The 0.18% score improvement is noise-level and comes entirely from reduced eval time (likely infrastructure variance), not from algorithmic progress. The plan's core hypothesis—that dequant-before-dot improves correctness—was falsified by the data.
* **Key Insight:** The parent's FP8 tensor-core path in GEMM1 was not the primary error source. Replacing it with FP32 dequant increased both absolute error (4096→8192) and relative error (3e7→1.4e9), proving that the FP8 dot with post-multiply scaling was actually more numerically stable than eager FP32 conversion. The real pathology remains in GEMM2's per-expert eager dequantization or in accumulated floating-point error from the scatter-add path.
* **Identified Risk:** This lineage is trapped in a correctness-first local minimum where every "safe" change either regresses performance or fails to move the needle on score. The 8192.0 absolute error ceiling suggests a structural mismatch (likely quantization boundary effects or routing weight precision loss) that cannot be fixed by swapping GEMM implementations alone.

**4. Actionable Guidance (The "What's Next?"):**
* **Recommend Stripping:** Revert the GEMM1 dequant-before-dot change. The parent's FP8 tensor-core path was superior in both speed and accuracy. Future descendants should preserve `tl.dot(fp8, fp8, out_dtype=tl.float32) * scale` for GEMM1.
