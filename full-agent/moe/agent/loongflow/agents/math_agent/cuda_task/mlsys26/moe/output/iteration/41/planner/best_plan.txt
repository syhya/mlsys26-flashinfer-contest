## Phase 1: Mathematical Definition

### 1.1 Variables
Let the implementation be parameterized by:
- `K_route`: routing algorithm implementation.
- `K1`: GEMM1 kernel implementation.
- `K2_base`: current-best GEMM2 kernel from `6da80c61`.
- `K2_tma`: new GEMM2 kernel with block-pointer/TMA-style W2 loads.
- `D`: dispatcher selecting between `K2_tma` and `K2_base`.
- `C2`: GEMM2 autotune config set.

The child keeps:
- `K_route = route_tokens from 6da80c61` unchanged.
- `K1 = grouped_gemm1_kernel / grouped_gemm1_kernel_tma setup from 6da80c61` unchanged.
- `K2 = {K2_base, K2_tma}` with deterministic guarded dispatch.

### 1.2 Constraints
The child must satisfy all hard constraints:

#### Routing constraints
For each token `t` and expert `e`:
- `q[t,e] = sigmoid(routing_logits[t,e])`
- `rank[t,e] = q[t,e] + routing_bias[e]`
- Partition experts into 8 groups of size 32.
- For each token/group, compute group score as sum of top-2 `rank` values within the group.
- Select top-4 groups.
- Mask all other experts.
- Select global top-8 experts from remaining masked `rank`.
- Routing weights must be normalized from `q` over those selected top-8 experts and then scaled by `routed_scaling_factor`.

#### GEMM1 constraints
For routed row `j` belonging to expert `e` and original token `tok[j]`:
- `G1[j,n] = Σ_k A[tok[j],k] * W1[e,n,k] * sA[k//128, tok[j]] * sW1[e,n//128,k//128]`
- Accumulation must be FP32.

#### SwiGLU constraints
For `i in [0, I)`:
- `x1 = G1[:, :I]`
- `x2 = G1[:, I:]`
- `C[:, i] = x1[:, i] * x2[:, i] * sigmoid(x2[:, i])`

#### GEMM2 constraints
For routed row `j` and hidden dim `h`:
- `O[j,h] = Σ_i C[j,i] * W2[e,h,i] * sW2[e,h//128,i//128]`
- Accumulation must be FP32.

#### Scatter constraints
- Final output is `output[tok[j], h] += routing_weight[j] * O[j,h]`
- Final dtype is `bfloat16` only at the output.

#### Safety constraints
- Any expert-base pointer arithmetic in Triton must use `int64`.
- `T` is dynamic and arbitrary.
- The implementation must remain self-contained in one Python file.

### 1.3 Objective
Minimize total latency:
- `f(X) = total_forward_time(X)`
subject to all correctness constraints above.

This is a **deterministic constrained model-selection problem** over a finite set of implementation choices, not a stochastic search problem.

## Phase 2: The Optimization Loop

### 2.1 Rebase onto `6da80c61` exactly
Use `6da80c61` as the full structural baseline, not the assigned older parent. Copy these components **verbatim**:
1. `_route_tokens` from `6da80c61`.
2. `grouped_gemm1_kernel`.
3. `grouped_gemm1_kernel_tma`.
4. `run()` structure, including:
   - early zero-return edge cases,
   - allocation strategy,
   - GEMM1 dispatch with try/except fallback,
   - bf16 epilogue (`O.to(torch.bfloat16)` then bf16 `index_add_`).
5. `grouped_gemm2_kernel` from `6da80c61` kept untouched as `K2_base`.

This gives a guaranteed strong baseline equal to the current best known solution.

### 2.2 Add a second GEMM2 kernel: `grouped_gemm2_kernel_tma`
Construct a new GEMM2 kernel whose math is identical to the current one, but whose W2 loads use `tl.make_block_ptr` and `tl.advance`.

The kernel should preserve the exact FP8 tensor-core pathway:
- load `x1`, `x2` from `G1`
- compute `c = x1 * x2 * tl.sigmoid(x2)`
- compute scalar quantization:
  - `c_abs_max = tl.max(tl.abs(c))`
  - `c_scale = tl.maximum(c_abs_max / 448.0, 1e-12)`
  - `c_fp8 = (c / c_scale).to(tl.float8e4nv)`
- load `w2_fp8` through a block pointer instead of raw pointer arithmetic
- do `dot_result = tl.dot(c_fp8, w2_fp8, out_dtype=tl.float32)`
- post-scale with `c_scale * w2_scale`
- multiply by routing weight
- store FP32 intermediate `O`

### 2.3 Mathematical equivalence of the new load path
The new kernel must describe each expert’s W2 slab as a 2D matrix with shape `(H, I)` using block pointers:
- base = `W2_ptr + pid_e.to(tl.int64) * stride_w2e`
- shape = `(H, I)`
- strides = `(stride_w2n, stride_w2k)`
- offsets = `(pid_n * BLOCK_N, 0)`
- block_shape = `(BLOCK_N, BLOCK_K)`
- order chosen so advancing across K is valid

Each iteration of the K-loop loads the exact same logical tile previously addressed by:
- `W2_ptr + pid_e * stride_w2e + n_offs[None,:] * stride_w2n + k_offs[:,None] * stride_w2k`

Since TMA changes only the data-movement mechanism, not the mathematical operands, the dot product is mathematically unchanged.

### 2.4 Add only a very small autotune expansion
Do **not** bloat autotune space aggressively.
Add at most one new config to the new GEMM2-TMA kernel beyond the inherited practical set, for example one `BLOCK_M=64 or 128` and `BLOCK_N=128` variant if needed.

The main search process is:
1. Keep `K2_base` as-is.
2. Add `K2_tma` with a compact config set.
3. In `run()`, try `K2_tma` first.
4. If compile/runtime launch fails, use `K2_base`.

This is effectively a deterministic **two-candidate model selection loop** with an exact fallback.

## Phase 3: The "Safety Valve" (CRITICAL)

### 3.1 Exact fallback mechanism
The code must include a deterministic fallback in `run()`:
- primary path: launch `grouped_gemm2_kernel_tma`
- fallback path: on any exception, launch the original `grouped_gemm2_kernel`

This guarantees:
- no crash,
- no correctness regression from a failed TMA experiment,
- performance floor approximately equal to `6da80c61`.

### 3.2 Preserve all fragile correctness-critical components
The following must remain unchanged from `6da80c61`:
1. Routing implementation.
2. GEMM1 kernels and dispatch.
3. GEMM2 FP8 quantization triad semantics.
4. bf16 epilogue.

### 3.3 Deterministic validity checks embedded by construction
The child must enforce these exact invariants:
- `tl.max(tl.abs(c))` has **no axis argument**.
- FP8 cast uses `tl.float8e4nv`.
- `tl.dot(c_fp8, w2_fp8, out_dtype=tl.float32)` uses FP8 on both inputs.
- `acc += dot_result * c_scale * w2_scale` stays in FP32.
- `pid_e.to(tl.int64)` is used before stride multiplication.
- `BLOCK_N=128` for GEMM2 so hidden-dim block scale indexing remains exact.
- output scatter remains outside Triton using bf16 `index_add_` exactly as in `6da80c61`.

### 3.4 Why this safety valve guarantees validity
This plan decouples:
- **Structure finding:** introduce a new load mechanism only in GEMM2.
- **Parameter tuning:** small finite config set handled by Triton autotune.
- **Hard constraint enforcement:** fallback to the exact already-successful implementation.

So even if the new structural variant underperforms or fails to compile, the returned program still satisfies the task and should retain `score >= 1.0`.
