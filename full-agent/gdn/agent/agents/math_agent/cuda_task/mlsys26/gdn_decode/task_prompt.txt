# Build Environment Constraints (CRITICAL)
The build environment uses `-D__CUDA_NO_BFLOAT16_CONVERSIONS__`, which **disables implicit casts** between `__nv_bfloat16` and `float`. You MUST use explicit conversion intrinsics:
- **BF16 to float**: `__bfloat162float(bf16_val)` -- NOT `(float)bf16_val`
- **float to BF16**: `__float2bfloat16(float_val)` -- NOT `(__nv_bfloat16)float_val`
- Always `#include <cuda_bf16.h>` at the top


# Correctness Requirements
1. Output shapes and dtypes must match exactly: `output` is bfloat16, `new_state` is float32.
2. Numerical tolerance: rtol=1e-2, atol=1e-2.
3. All intermediate accumulation must be in **FP32**; convert back to BF16 only for final `output` store.
4. All variable axes (e.g., `batch_size`) must work for any valid value.
5. Gate formulas must be exact:

# Core Optimization Strategy
Your solution must demonstrate a **substantial architectural or performance improvement** over a naive implementation. Superficial changes are forbidden. The following dimensions are key areas where meaningful gains can be achieved — addressing more dimensions generally leads to better performance:

## 1. Thread/Block Parallelism Design
The GDN decode state update is dominated by reads and writes to the large state tensor `[B, HV, V, K]`.
The optimal work partitioning across warps and thread blocks is a key design decision:
- Threads can be mapped to the K dimension (128 elements), V rows, or a combination of both.
- Block size and grid shape directly affect occupancy and warp utilization.
- Different batch sizes may benefit from different launch configurations to maximize GPU utilization.

## 2. Memory Access Optimization
The state tensor `[B, HV, 128, 128]` in float32 is the dominant memory traffic source.
Effective strategies include:
- Vectorized loads/stores (e.g., `float4` for float32 state rows, `uint2` / `__nv_bfloat162` for bfloat16 inputs) and other wide-load approaches to maximize memory bus utilization.
- Keeping q and k vectors in registers across the V-row loop to avoid repeated global memory loads.
- Fusing state decay (`g * h`) and state update (`+ dv * k`) into a single read-modify-write pass to minimize memory traffic.

## 3. Warp-level Computation
Dot products over the K=128 dimension (kh, qh, qk_dot) are reduction operations across a warp.
Effective strategies include:
- Warp shuffle primitives such as `__shfl_xor_sync` / `__shfl_sync`, or other intra-warp communication mechanisms, for fast reductions.
- Fusing multiple reductions (e.g., kh and qh) into a single communication step to save instructions.
- Broadcasting per-head scalar values (g, beta) from a single lane to all lanes in the warp.

## 4. Scalar Hoisting and Computation Order
Gates `g` and `beta` are per-head scalars independent of V and K indices.
Reordering computation can significantly reduce instruction count and register pressure:
- Values independent of the V-row loop (gates, qk_dot, scale factors) can be precomputed once before the loop begins.
- The output formula `scale * (qh + dv * qk_dot)` can be algebraically rearranged to reduce operations in the inner loop.

## 5. Instruction-level Efficiency
The gate formulas (softplus, sigmoid, exp) and inner-loop accumulations are hot paths.
Fast math intrinsics (e.g., `__expf`, `__logf`, `__frcp_rn`) and fused multiply-add operations (e.g., `fmaf`) reduce instruction latency on CUDA hardware. Using `__launch_bounds__` helps the compiler make better register allocation decisions and improve occupancy.

## 6. Library Usage
You are encouraged to call third-party CUDA libraries directly from `kernel.cu` when they provide a performance advantage. Available libraries in the build environment include:
- **CUB** (`<cub/cub.cuh>`): block-level and warp-level primitives (e.g., `cub::WarpReduce`, `cub::BlockReduce`) for dot-product reductions over the K dimension.
- **CUTLASS / CuTe** (`<cute/tensor.hpp>` etc.): tile-level abstractions and layout utilities; less central for GDN's element-wise state update, but useful if a tiled approach is explored.
- **Standard CUDA math** (`<cuda_bf16.h>`, `<cuda_fp16.h>`): bfloat16/fp16 arithmetic helpers already available without extra includes.
