The Planner correctly identified a mathematical shortcut from the task spec ("set of indices must be correct, ordering arbitrary") that makes scoring/sorting *provably unnecessary* when `L ≤ K`. The Executor implemented it as a clean partition: each batch is handled by exactly one path, eliminating any race risk.
*   **Key Insight:** **Reading the task specification for mathematical shortcuts beats any micro-optimization.** The previous iteration line spent generations chiseling the scoring kernel (uint4 loads, transposed Q, etc.) for ~10% gains; a single specification-driven dispatch yielded a 2× win. *   **Identified Risk:** The win is workload-distribution-dependent — gains are large only because the benchmark contains many short-sequence batches. For workloads where all batches have `L > 2048`, this collapses to parent performance. Additionally, the long-path still pays the full O(N) CUB segmented sort cost, which is now the unambiguous bottleneck for long sequences.
