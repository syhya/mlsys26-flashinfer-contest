## Phase 1: Mathematical Definition

### 1.1 Task model
For each token `t` and head `h`, define
- Query vector
  - `q = concat(q_nope[t,h,:], q_pe[t,h,:]) in R^576`
- Sparse key/value candidates indexed by `sparse_indices[t, j]`, `j = 0..2047`
- For a valid sparse index `idx >= 0`:
  - `page = idx >> 6`
  - `offset = idx & 63`
  - `k_nope_j = ckv_cache[page, offset, :] in R^512`
  - `k_pe_j   = kpe_cache[page, offset, :] in R^64`
  - `logit_j = sm_scale * (dot(q_nope[t,h,:], k_nope_j) + dot(q_pe[t,h,:], k_pe_j))`
- Output:
  - `out[t,h,:] = sum_j softmax(logit)_j * k_nope_j`
- Base-2 log-sum-exp:
  - `lse[t,h] = log2(sum_j exp(logit_j))`

Padding rule:
- If `sparse_indices[t,j] == -1`, that position is masked and must not contribute to dot products, softmax normalization, or output.
- If all 2048 entries are `-1`, then:
  - `out[t,h,:] = 0`
  - `lse[t,h] = -INF`

### 1.2 Variables, constraints, objective

#### Variables `X`
1. Dispatch threshold `tau` separating:
   - fused low-latency path
   - fused throughput path
2. Tile size `Bk` over sparse positions, chosen from a small deterministic set, with primary target `Bk = 32`.
3. Per-(token, head) online softmax state:
   - running max `m`
   - running sum `l`
   - unnormalized output accumulator `O in R^512`
4. Shared-memory staging buffers:
   - `Kc_tile[Bk, 512]`
   - `Kp_tile[Bk, 64]`
   - `idx_tile[Bk]`

#### Constraints `C`
1. **Exact masked semantics**: only valid `idx >= 0` positions contribute.
2. **FP32 accumulation**: all dot products, `m/l` updates, and output accumulation must be float32.
3. **Final BF16 only**: convert to BF16 only when writing final output.
4. **Base-2 LSE**:
   - `lse = m * log2(e) + log2(l)` when `l > 0`
5. **All-invalid guard**:
   - if no valid index seen, write zeros and `-INF`.
6. **Build constraint**:
   - only explicit BF16 conversion intrinsics may be used.
7. **Kernel resource constraint**:
   - reduce shared memory and register pressure enough to keep occupancy competitive.

#### Objective `f(X)`
Minimize
- total runtime = launch overhead + gather/load cost + dot-product cost + softmax/update cost + final store cost
subject to all correctness constraints above.

### 1.3 Solver strategy
Use a **deterministic exact recurrence solver** rather than stochastic search:
- core recurrence = FlashAttention-style online softmax
- runtime path selection = deterministic dispatch based on `num_tokens`
- kernel topology search = tiny discrete search over a few safe launch shapes and tile sizes, not random exploration

This satisfies the planner requirement to rely on mathematics instead of luck.

## Phase 2: The Optimization Loop

### 2.1 Overall algorithmic structure
Refactor the parent into a **two-path fused design**:

#### Path A: low-latency fused path for small workloads
- Replace the current 2-kernel split-K route for small `num_tokens` with a **single-pass fused kernel**.
- Use grid decomposition that increases block count for small workloads without requiring any cross-block reduction.
- Primary mapping:
  - one block handles one `(token, head)` pair
  - block size should be reduced from `32 x 16 = 512` threads to a lower-risk shape such as `128` or `256` total threads, with each block covering fewer heads or exactly one head
- Goal: avoid the parent’s low-workload HBM round-trip and under-filled launches.

#### Path B: throughput fused path for medium/large workloads
- Keep the parent’s one-pass online kernel idea, but refactor memory staging and thread mapping to reduce register pressure and improve occupancy.
- Preserve the already successful pieces:
  - register-cached query conversion to FP32
  - `fmaf`-based accumulation
  - vectorized global reads when aligned

### 2.2 Exact online softmax recurrence to use in every path
For each valid key with score `x`:
- `m_new = max(m, x)`
- `alpha = exp(m - m_new)` if prior valid key exists else `0`
- `beta  = exp(x - m_new)`
- `l_new = l * alpha + beta`
- `O_new = O * alpha + beta * V`
- then assign `m = m_new`, `l = l_new`, `O = O_new`

At end:
- if `l > 0`, `out = O / l`
- `lse = m * 1.4426950408889634f + log2f(l)`
- else zero output and `-INF`

This is the exact mathematical invariant. Executor should preserve it across all refactors.

### 2.3 Concrete optimization steps

#### Step 1: Change small-workload mapping to remove split-K workspace
- Eliminate allocation of:
  - `O_tmp`
  - `m_tmp`
  - `l_tmp`
- Replace with a single fused kernel for small `num_tokens`.
- Suggested mapping:
  - `grid.x = num_tokens`
  - `grid.y = 16` heads
  - `block.x = 32` or `64` or `128`
- If using `block.x = 32`, one warp handles one head and one block handles one `(token, head)`.
- This creates many more blocks than the parent’s split-K reduce path for tiny workloads and completely removes the intermediate global-memory write/read cycle.

#### Step 2: Reduce shared-memory footprint
The parent allocates shared memory for 32 gathered keys:
- `smem_Kc[32 * 512]`
- `smem_Kp[32 * 64]`
This is large and contributes to occupancy pressure.

Refine it by using a smaller per-stage gather tile when needed:
- test `Bk = 16` for the low-latency path
- keep `Bk = 32` for the throughput path

Deterministic micro-search rule:
- low-latency path uses `Bk = 16`
- throughput path uses `Bk = 32`

This is not random; it is a fixed structural split.

#### Step 3: Make loads page-aware and order-preserving
For each gathered sparse index:
- decode `page = idx >> 6`, `offset = idx & 63`
- load from:
  - `ckv_cache[(page * 64 + offset), :]`
  - `kpe_cache[(page * 64 + offset), :]`
This avoids assuming flat layout in a way that could hide indexing bugs.

The executor should explicitly compute the row pointer as:
- `row = ((int64_t)page) * 64 + offset`
then index contiguous cache tensors by `row`.

#### Step 4: Preserve vectorization but only where alignment is safe
- Continue using vectorized loads (`float4`/`int4`) for BF16 cache reads only when pointers are aligned and layout is contiguous.
- Do **not** force risky reinterpret casts everywhere.
- Fallback to scalar/pair (`__nv_bfloat162`) loads if alignment is uncertain.

This is a robustness-first optimization.

#### Step 5: Use warp-level reduction for logits only
- Keep warp shuffle reduction for the dot product.
- Avoid block-wide reductions or heavy synchronization in the core score path.
- If one warp owns one head, the reduction is naturally isolated and low-overhead.

#### Step 6: Hoist query conversion fully out of the sparse loop
This was already a good parent optimization and must be preserved:
- convert `q_nope` and `q_pe` from BF16 to FP32 once per `(token, head)`
- store fragments in registers
- never re-convert inside the inner sparse-key loop

#### Step 7: Separate output accumulation fragments to control register pressure
The parent stores all `512` output dimensions as `8 x float2` fragments per lane.
Keep this structure, but if register count remains too high:
- split output accumulation into two deterministic halves of 256 dims each inside the same kernel
- reuse registers across the two halves
- only if needed to lower occupancy pressure

This is a structure/parameter decoupling:
- structure = same one-pass exact recurrence
- parameter tuning = how many output dims are live in registers at once

### 2.4 Deterministic dispatch rule
Use a simple robust rule in `dsa_forward`:
- `num_tokens <= 32`: low-latency fused single-head-per-block kernel
- `num_tokens > 32`: throughput fused kernel based on the parent’s current one-pass design

If the executor finds a slightly different threshold easier to implement, it may use one nearby fixed threshold such as `64`, but it must remain deterministic and simple.

The global perspective shows best score `4.5065` and average `3.508`, so the island is still improving, not stuck. Therefore we should **vertically optimize** the current successful family instead of switching to a totally different algorithm.

## Phase 3: The "Safety Valve" (CRITICAL)

This section is mandatory. The child must include deterministic repair/guard logic that guarantees validity even if an optimization path underperforms.

### 3.1 Host-side validation and canonicalization
Before launch, enforce:
1. all tensors on CUDA
2. contiguous layout via `.contiguous()` if needed
3. dtypes exactly:
   - BF16 for `q_nope`, `q_pe`, `ckv_cache`, `kpe_cache`
   - int32 for `sparse_indices`
4. shape checks:
   - `q_nope[...,512]`
   - `q_pe[...,64]`
   - cache last dims `512` and `64`
   - `sparse_indices.shape[1] == 2048`

If any check fails, use `TORCH_CHECK` and abort deterministically.

### 3.2 All-invalid deterministic repair
Inside every kernel, track a boolean or integer `valid_count` per `(token, head)`.
- increment only when `idx >= 0`
- after processing all keys:
  - if `valid_count == 0`, write
    - output zeros
    - `lse = -INFINITY`
  - skip normalization

This prevents NaNs and guarantees compliance with the task statement.

### 3.3 Exact base-2 lse enforcement
Do not rely on any implicit identity.
Always compute at the end:
- if `l > 0`: `lse = m * 1.4426950408889634f + log2f(l)`
- else `lse = -INFINITY`

### 3.4 Explicit BF16 conversion enforcement
Because the environment uses `-D__CUDA_NO_BFLOAT16_CONVERSIONS__`, the executor must ensure:
- BF16 -> float uses `__bfloat162float()` or `__bfloat1622float2()`
- float -> BF16 uses `__float2bfloat16()` or `__floats2bfloat162_rn()`
- no C-style or implicit casts are present anywhere

### 3.5 Deterministic fallback path in host code
If implementing the new small-workload fused kernel proves too risky, keep a compile-safe fallback:
- if `num_tokens <= tau` and a guard condition fails (e.g. unsupported launch shape, temporary debug macro, or non-contiguous input after checks), call the parent’s already-correct one-pass kernel instead of the split-K path
- do **not** use experimental atomic sync as the only low-token implementation

The goal is score protection first, speed second.
