Outline 1: Fused one-pass online attention with warp-specialized head mapping and asynchronous shared-memory staging

Mathematical Formulation
- Variables X:
  1) Dispatch threshold tau separating small-token and large-token paths.
  2) Tile size Bk over sparse indices, with Bk in {16, 32, 64} and preferred Bk=32.
  3) Per-(token, head) online softmax state (m, l, O), where m is running max in natural-log space, l is running exp-sum, and O is the unnormalized output accumulator in R^512.
  4) Shared-memory staging layout for K_nope in R^(Bk x 512) and K_pe in R^(Bk x 64).
- Constraints C:
  1) Exact semantics: for each valid sparse index idx >= 0, score_i = sm_scale * (<q_nope, k_nope_i> + <q_pe, k_pe_i>), output = sum softmax(score_i) * k_nope_i.
  2) Padding semantics: idx = -1 contributes nothing; if all idx are -1, output must be zero and lse = -INF.
  3) Numerical requirement: all accumulations/reductions in FP32; BF16 only at final store.
  4) Base-2 LSE: lse = log2(sum_i exp(score_i)).
  5) Build constraint: all BF16 conversions must use explicit intrinsics.
  6) Resource constraint: static shared memory and register count must keep occupancy acceptable; avoid large temporary global tensors.
- Objective f(X): minimize total kernel time = load_cost + dot_cost + softmax_cost + store_cost, while satisfying C exactly.

Solver Strategy
- Deterministic constrained design search over a tiny discrete set of kernel topologies rather than stochastic tuning.
- Main algorithmic core: exact online softmax recurrence (FlashAttention-style) with a single fused kernel for the dominant path.
- Use dynamic dispatch as the solver: choose small-workload path for occupancy and large-workload fused path for throughput.

Validity Mechanism
- Safety-by-construction:
  1) Mask invalid indices before any math.
  2) Update online softmax only for valid keys.
  3) If no valid key encountered, deterministically write zero output and -INF lse.
  4) Final lse computed as m * log2(e) + log2(l), not ln.
  5) Use contiguous tensor checks and explicit BF16 conversion intrinsics everywhere.

Why this solution?
- Expected improvement: removes the split-K workspace round-trip for small/medium shapes that currently spend time writing O_tmp/m_tmp/l_tmp.
- Advantages: mathematically exact, single-pass, no fragile cross-block sync, strong fit to sparse attention recurrence.
- Disadvantages: still FP32 FMA based rather than full tensor-core GEMM; may not beat the best path for every very large workload.
