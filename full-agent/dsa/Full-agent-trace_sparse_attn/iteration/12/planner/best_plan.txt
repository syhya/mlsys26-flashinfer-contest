## Phase 1: Mathematical Definition

### 1.1 Task model
For each token `t` and head `h`, define the query
- `q_nope[t,h] in R^512`
- `q_pe[t,h] in R^64`

Each sparse candidate position `j in {0,...,2047}` is encoded by
- `idx = sparse_indices[t,j]`
- if `idx == -1`, this entry is invalid and must contribute nothing
- else:
  - `page = idx >> 6`
  - `offset = idx & 63`
  - `row = page * 64 + offset`
  - `k_nope_j = ckv_cache[page, offset, :] in R^512`
  - `k_pe_j   = kpe_cache[page, offset, :] in R^64`

The score before softmax is
- `s_j = dot(q_nope[t,h], k_nope_j) + dot(q_pe[t,h], k_pe_j)`
- `x_j = sm_scale * s_j`

The output is
- `out[t,h,:] = sum_j softmax(x)_j * k_nope_j`
where the sum runs only over valid indices.

The required log-sum-exp output is
- `lse[t,h] = log2(sum_j exp(x_j))`
again over valid indices only.

If all sparse entries are invalid, the required exact behavior is
- `out[t,h,:] = 0`
- `lse[t,h] = -INF`

### 1.2 Variables, constraints, and objective

#### Variables `X`
1. Dispatch threshold `tau` for choosing the low-latency vs throughput kernel path.
2. Sparse tile size `Bk`, with deterministic candidate values `{16, 32}`.
3. Per `(t,h)` online softmax state:
   - running max `m`
   - running denominator `l`
   - unnormalized accumulator `O in R^512`
4. Shared-memory staged buffers for a tile of sparse keys:
   - `idx_tile[Bk]`
   - `Kc_tile[Bk,512]`
   - `Kp_tile[Bk,64]`

#### Constraints `C`
1. **Masked exactness**: entries with `idx == -1` must not affect dot products, softmax normalization, output, or `lse`.
2. **FP32 accumulation**: all intermediate math must be float32:
   - dot products
   - online softmax max/sum updates
   - output accumulation
3. **BF16 store only at the end**: output tensor is BF16, but only final writeback may convert FP32 to BF16.
4. **Base-2 LSE**: final `lse` must be exactly
   - `m * log2(e) + log2(l)` when `l > 0`
5. **All-invalid repair**: if no valid entry is observed, output zeros and `-INF`.
6. **Build compatibility**: only explicit BF16 intrinsics may be used because implicit BF16 conversions are disabled.
7. **Polynomial-time implementation**: runtime must remain linear in the number of sparse entries and feature dimensions.

#### Objective `f(X)`
Minimize end-to-end runtime
- `f(X) = launch_overhead + sparse_gather_cost + dot_cost + online_softmax_cost + final_store_cost`
subject to all correctness constraints above.

### 1.3 Mathematical solver view
This is not a random search problem. The correct “solver” is a **deterministic exact recurrence**:
- use FlashAttention-style online softmax to process valid keys in one pass
- use deterministic dispatch over a tiny finite parameter set
- structurally enforce validity by skipping invalid entries before arithmetic

This satisfies the requirement to rely on math rather than luck.

## Phase 2: The Optimization Loop

### 2.1 Main optimization direction
The child should keep the current parent’s successful core idea:
- one-pass online softmax
- query vectors hoisted into registers
- warp-level dot-product reduction
- FP32 recurrence and BF16 final store

But it should **replace the small-workload split-K path** with a **single fused low-latency kernel**.

So the optimization loop is:
1. Preserve the current high-throughput fused kernel family for medium/large workloads.
2. Remove the temporary global-memory split-K path for small workloads.
3. Introduce a low-latency fused kernel that computes a full `(token, head)` result in one pass.
4. Use simple deterministic dispatch between the two.

### 2.2 Exact online softmax recurrence
For each valid logit `x` and value vector `V = k_nope_j`, maintain:
- `m_new = max(m, x)`
- `alpha = 0` if `m == -INF`, otherwise `exp(m - m_new)`
- `beta = exp(x - m_new)`
- `l_new = l * alpha + beta`
- `O_new = O * alpha + beta * V`

Then assign:
- `m = m_new`
- `l = l_new`
- `O = O_new`

At the end:
- if `l > 0`, `out = O / l`
- `lse = m * 1.4426950408889634f + log2f(l)`
- else use the all-invalid repair branch

This recurrence is exact and should be reused unchanged in every kernel path.

### 2.3 Concrete child-generation steps

#### Step 1: Delete the default split-K path for small workloads
The parent currently uses, for small token counts:
- `split_k_compute_kernel`
- `split_k_reduce_kernel`
- temporary global tensors `O_tmp`, `m_tmp`, `l_tmp`

The child should remove this from the default dispatch path.

Reason:
- even if fast, it still performs an unnecessary HBM round-trip
- it increases launch overhead
- it is structurally less elegant than a fused one-pass kernel

#### Step 2: Introduce a fused low-latency kernel
Add a new kernel, conceptually:
- `dsa_forward_small_kernel(...)`

Recommended mapping:
- `grid.x = num_tokens`
- `grid.y = 16`
- one block computes one `(token, head)` pair

Recommended block shape:
- start from `block.x = 32` or `64`
- prefer one warp per `(token, head)` if possible

Reason:
- this maximizes block count for small `num_tokens`
- avoids inter-block communication completely
- preserves exact one-pass semantics

#### Step 3: Keep the throughput path as a refined version of the parent’s fused kernel
For larger workloads, retain the parent’s fused kernel structure because it is already scoring very highly.

Keep these parent strengths:
1. query BF16 -> FP32 conversion hoisted outside the sparse loop
2. warp-level shuffle reduction for logits
3. vectorized BF16 loads where safe
4. per-lane register fragments for output accumulation

Only refine if safe:
- reduce unnecessary synchronization
- slightly lower register pressure if occupancy is harmed
- keep exact logic unchanged

#### Step 4: Deterministic finite dispatch, not heuristic randomness
Use a simple threshold:
- if `num_tokens <= tau`, launch `dsa_forward_small_kernel`
- else launch the throughput fused kernel

Candidate threshold:
- `tau = 32` as first choice
- `tau = 64` as acceptable alternate if implementation is simpler

This is a deterministic discrete choice, not random tuning.

#### Step 5: Sparse tile sizing
Use a fixed tile size per path:
- low-latency path: `Bk = 16`
- throughput path: `Bk = 32`

Reason:
- lower shared-memory footprint in the small path
- preserve throughput-friendly staging in the large path

This is a tiny finite parameter selection and is easy to implement safely.

#### Step 6: Explicitly structure index decoding
Use a helper device function such as:
- `decode_idx(int idx, int& page, int& offset)`

Then compute:
- `page = idx >> 6`
- `offset = idx & 63`
- `row = page * 64 + offset`

Although `row == idx` numerically, keeping explicit page/offset decoding matches task semantics and reduces indexing mistakes.

#### Step 7: Preserve vectorization conservatively
Use vectorized loads only where alignment is naturally safe:
- `float4` / `int4` or paired BF16 loads for cache rows
- `__nv_bfloat162` for query and key pair fragments

Do not over-force reinterpret casts in uncertain cases.
A safe fallback to pairwise BF16 loads is preferable to a brittle aggressive cast.

#### Step 8: Control register pressure deterministically
If the small fused kernel becomes register-heavy, split the output accumulation logically into fixed halves:
- accumulate dims `[0,255]`
- then dims `[256,511]`
within the same kernel and same recurrence state

Only do this if needed for occupancy.
This preserves exactness because the normalization state `(m,l)` is unchanged.

## Phase 3: The "Safety Valve" (CRITICAL)

This section is mandatory. The child must include deterministic validity enforcement so the result remains correct even if some optimization detail underperforms.

### 3.1 Host-side validation
At entry to `dsa_forward`, enforce with `TORCH_CHECK`:
1. all tensors are CUDA tensors
2. all tensors are contiguous
3. dtypes are exact:
   - BF16 for `q_nope`, `q_pe`, `ckv_cache`, `kpe_cache`
   - int32 for `sparse_indices`
4. shapes are exact:
   - `q_nope` = `[T,16,512]`
   - `q_pe` = `[T,16,64]`
   - `ckv_cache` = `[P,64,512]`
   - `kpe_cache` = `[P,64,64]`
   - `sparse_indices` = `[T,2048]`
5. token/page dimension consistency holds

If any check fails, abort deterministically.

### 3.2 All-invalid deterministic repair branch
Inside every kernel, keep a `valid_count` or equivalent boolean.
- increment only for `idx >= 0`
- after all sparse positions are processed:
  - if `valid_count == 0` or `l <= 0`, write zero BF16 output and `-INF` for `lse`

This prevents NaNs and ensures strict compliance with task semantics.

### 3.3 Base-2 LSE enforcement
Always compute final `lse` as:
- if `l > 0`: `lse = m * 1.4426950408889634f + log2f(l)`
- else `lse = -INFINITY`

Do not use natural-log output and do not rely on any implicit conversion identity.

### 3.4 BF16 conversion safety
Because the build uses `-D__CUDA_NO_BFLOAT16_CONVERSIONS__`, all BF16 handling must use explicit intrinsics only:
- BF16 -> FP32:
  - `__bfloat162float(...)`
  - `__bfloat1622float2(...)` or equivalent supported paired intrinsic
- FP32 -> BF16:
  - `__float2bfloat16(...)`
  - `__floats2bfloat162_rn(...)`

The child must not introduce any implicit BF16 cast.

### 3.5 Fallback kernel policy
If the new small fused kernel is not fully stable during implementation, the deterministic safety fallback is:
- use the parent’s already-correct fused one-pass kernel for all sizes
- do **not** keep split-K as the only small-token path unless necessary for correctness protection

This protects score first.

### 3.6 No risky cross-block synchronization
Do not use:
- atomics for normalization merging
- cooperative grid sync
- deadlock-prone inter-block protocols

All normalization should remain local to one block in the small path and structurally independent in the large path.
