**3. Strategic Analysis (The "So What?"):**
- **Root Cause:** Primarily a plan-quality failure, with a subtle correctness-adjacent execution defect. The plan repeated the earlier-falsified layout-permutation idea (sibling `9925fb3e` already regressed on the same permutation) — the producer still parallelizes heads via `threadIdx.y`, so 16 warps writing simultaneously under the new layout are scattered across 128KB-stride slabs instead of contiguous per-(t,s) slabs, hurting write coalescing. Additionally, the early-skip path has a latent bug: when `partition_valid==0`, all 512 threads `return`, but only `(lane==0)` writes `m_tmp`/`l_tmp` for its own `h`. Because every head's warp in the block participates in the same `partition_valid` decision and each warp's lane-0 writes its own `(t,h,s)` neutral entry, correctness is preserved — but it is preserved by coincidence of the block-level head mapping, not by design clarity. If any future refactor moved head-parallelism out of `threadIdx.y`, this path would silently produce undefined `m_tmp`/`l_tmp`. The kernel is also demonstrably faster in absolute terms (0.0228 ms vs 0.0261 ms parent) — the regression is entirely a reference-baseline artifact (2.66 ms vs 3.03 ms), not a true kernel slowdown.
- **Key Insight:** The most important lesson is that score deltas in this lineage are now dominated by reference-baseline variance (12–18% swings across runs), not kernel improvements. This child's *absolute kernel latency is the best observed in the lineage*, yet it scores below parent purely because the reference ran 12% faster on this execution. Salvageable concepts: (1) the partition-level early-skip idea is promising and appears to have contributed to the absolute latency drop, and (2) the indexing helpers (`otmp_idx`, `ml_idx`) improve code hygiene for future workspace experiments.
- **Identified Risk:** The lineage has now regressed *four consecutive times* on small-path workspace experiments from parent `bad043e5` (BF16 dtype, S=32 partitions, layout permutation twice). The Evolux planner is stuck in a failure mode: repeatedly proposing workspace micro-tweaks on a parent that is already suboptimal relative to its cousin `94736692` (12.63), which has never been adopted as the exploitation baseline. Secondary risk: the `correctness=1.0` gate is proving insufficient — max_rel_error varies wildly (0.30 to 793) across siblings with identical pass flags, and score movements driven by reference-baseline noise may be misread as real progress or regression.

Any future layout exploration must first change producer parallelism (one block per (t,h,s) instead of (t,s) with head-in-threadIdx.y).

Prototype a fused single-kernel split-K with cooperative-groups grid sync OR a block-per-(t,h) design where 16 warps cooperatively reduce 64 partitions in shared memory — this removes the `O_tmp` HBM round-trip entirely instead of negotiating its layout.
