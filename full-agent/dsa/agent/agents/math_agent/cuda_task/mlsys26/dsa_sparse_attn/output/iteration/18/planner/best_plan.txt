- Inputs:

## Phase 1: Mathematical Definition

### 1.1 Core mathematical model
For each token `t` and head `h`, define the sparse key list:
- `idx_s = sparse_indices[t, s]` for `s in [0, 2048)`
- Valid entries satisfy `idx_s != -1`
- Invalid entries must contribute exactly zero to the output and must not affect softmax or `lse`

For a valid `idx_s`, define:
- `Kc_s = ckv_cache[idx_s, :]` reshaped as length-512 vector
- `Kp_s = kpe_cache[idx_s, :]` reshaped as length-64 vector
- `Qn = q_nope[t,h,:]`
- `Qp = q_pe[t,h,:]`

Then the attention logit is:
- `raw_s = dot(Qn, Kc_s) + dot(Qp, Kp_s)`
- `z_s = sm_scale * raw_s`

The reference output is:
- `weight_s = exp(z_s)` for valid indices only
- `L = sum_s weight_s`
- `O = (1 / L) * sum_s weight_s * Kc_s`

The required `lse` is **base-2**:
- `lse = log2(L)`

### 1.2 Online softmax recurrence
To avoid storing logits or intermediate matrices, use the FlashAttention-style online recurrence in FP32.
For each valid sparse entry processed in sequence, maintain:
- `m`: running maximum of scaled logits
- `l`: running sum of exponentials in the shifted frame
- `A[d]`: running unnormalized output accumulator for each output dimension `d in [0,512)`

Update rule for a new scaled logit `z` and value vector `V = Kc_s`:
- `m_new = max(m, z)`
- `alpha = exp(m - m_new)` if prior state valid else `0`
- `beta = exp(z - m_new)`
- `l_new = l * alpha + beta`
- `A_new[d] = A[d] * alpha + beta * V[d]`
- `m = m_new`, `l = l_new`

Final normalized output:
- `O[d] = A[d] / l` if `l > 0`, else `0`

Final base-2 log-sum-exp:
- `lse = m * log2(e) + log2(l)` if `l > 0`, else `-inf`

### 1.3 Variables, constraints, objective
#### Variables `X`
- Dispatch threshold `tau_small` controlling small/large workload path
- Thread-block decomposition over token/head work
- Shared-memory staging buffers for one sparse tile of 32 entries
- Online state `(m, l, A[512])` in FP32 per `(t,h)`

#### Constraints `C`
1. `idx == -1` must be treated as masked and must not affect `m`, `l`, `A`, output, or `lse`.
2. All dot-product accumulation must be FP32.
3. All online-softmax state must be FP32.
4. Output must be BF16 only at final store.
5. `lse` must be computed in **base 2**, not natural log.
6. Explicit BF16 conversion intrinsics must be used due to `-D__CUDA_NO_BFLOAT16_CONVERSIONS__`.
7. Shapes and dtypes must match exactly.

#### Objective `f(X)`
Minimize total runtime cost:
- `f(X) = HBM_read_cost + HBM_write_cost + launch_overhead + stall_cost`
subject to exact semantics and output validity.

The highest-leverage term is the parent’s low-workload `HBM_write_cost`, coming from FP32 temporary workspaces. The new child must reduce that term by eliminating split-K temporary storage.

## Phase 2: The Optimization Loop

This task is not a stochastic optimization problem; the “optimization loop” here is a deterministic design loop over architecture choices. The implementation should evaluate a small, fixed family of kernel organizations and encode the best one directly.

### 2.1 Candidate structures to consider
Implement the child around two deterministic execution paths.

#### Path A: Small-workload persistent fused kernel
Use this when `num_tokens` is small and the parent currently underfills the GPU.
- Use a persistent CTA or grid-stride scheduler over `(token, head)` jobs.
- Each warp computes one head at a time.
- A CTA may own 4, 8, or 16 warps depending on register pressure.
- Each warp performs the full 2048-sparse-entry online recurrence for one `(t,h)` without writing intermediates to HBM.

#### Path B: Large-workload fused kernel
Use this when `num_tokens` is already large enough to saturate the GPU.
- Keep the parent’s general one-pass fused approach.
- Retain vectorized loads and warp-shuffle reduction.
- Simplify if needed to reduce registers and shared memory.
- Do **not** allocate temporary split-K workspaces.

### 2.2 Deterministic finite search over kernel structures
The executor should not perform runtime autotuning, but it should implement the code in a way that reflects a finite “search” over robust configurations.

Candidate settings:
- Warps per CTA in small path: `{4, 8}`
- Sparse tile size: fixed `32` entries per tile
- Output register blocking per lane:
  - Option 1: each lane owns 16 output BF16 values = 8 `float2`
  - Option 2: each lane owns 32 output BF16 values = 16 `float2` if registers allow
- Dispatch threshold `tau_small`: start with `num_tokens < 192` or `num_tokens < 256`

Selection rule:
- Prefer the smallest-register version that still keeps the output accumulator fully in registers.
- If register pressure becomes too high, split output accumulation into two 256-d passes inside the same fused kernel, still avoiding HBM temporaries.

### 2.3 Concrete kernel algorithm for the small-workload path
For each `(t,h)` job:

1. **Load queries into registers**
   - Load `q_nope[t,h,:]` into per-lane register fragments using paired BF16 loads.
   - Load `q_pe[t,h,:]` similarly.
   - Convert all BF16 fragments to FP32 immediately using explicit intrinsics.

2. **Tile sparse indices by 32**
   - For `tile = 0..63`, load `sparse_indices[t, tile*32:(tile+1)*32]` into shared memory.
   - This shared tile becomes the structural mask for the next compute step.

3. **Stage key/value data into shared memory**
   - For every valid entry in the 32-entry tile, cooperatively load:
     - `ckv_cache[idx, :]` into shared memory tile `smem_kc[32][512]`
     - `kpe_cache[idx, :]` into shared memory tile `smem_kp[32][64]`
   - Invalid entries may either be skipped or zero-filled; skipping is preferred if the compute loop checks validity.
   - Use 128-bit vectorized loads (`int4`/`float4` reinterpret pattern) wherever alignment allows.

4. **Compute logits and update online state**
   - For each valid row `i` in the sparse tile:
     - Compute partial dot products per lane for the 512-d `q_nope · Kc_i` term.
     - Compute partial dot products per lane for the 64-d `q_pe · Kp_i` term.
     - Reduce within warp using `__shfl_down_sync`.
     - Lane 0 obtains the full FP32 `raw_i`, then forms `z_i = sm_scale * raw_i`.
     - Broadcast `z_i` to warp.
     - Update `(m,l)` and the FP32 output accumulator `A` using the online recurrence.

5. **Finalize**
   - After all 2048 slots are processed, normalize `A` by `l`.
   - Convert final FP32 outputs to BF16 using `__float2bfloat16` or `__floats2bfloat162_rn`.
   - Store directly to output tensor.
   - Lane 0 stores `lse = m * log2(e) + log2(l)`.

### 2.4 Concrete kernel algorithm for the large-workload path
For large `T`, preserve the parent’s existing advantage: good occupancy and one-pass compute.

Required modifications:
1. Remove the split-K workspace path entirely unless a truly necessary fallback is needed.
2. Use the same fused online-softmax recurrence as above.
3. Keep shared-memory staging of `ckv` and `kpe` by 32-entry sparse tiles.
4. Keep vectorized global reads.
5. If beneficial, reduce shared memory footprint to allow more resident blocks.

### 2.5 Multi-start mandate
There is **no randomness** in this plan, so no stochastic multi-start loop is required. The deterministic analog is the finite candidate-set design loop above.

## Phase 3: The "Safety Valve" (CRITICAL)

This phase is mandatory. It ensures the final code cannot silently become invalid while chasing speed.

### 3.1 Hard correctness repair rules
Implement the following deterministic safeguards in the kernel and dispatcher.

#### Safety Rule 1: Mask invalid sparse entries before use
For every sparse index:
- If `idx == -1`, the kernel must skip both loads and recurrence update.
- This guarantees padding contributes exactly zero.

#### Safety Rule 2: Exact FP32 accumulation
All of the following must be FP32:
- dot-product partial sums
- warp-reduced logits
- online softmax state `m`, `l`
- output accumulator `A[512]`

This avoids the main class of silent numerical drift.

#### Safety Rule 3: Deterministic zero-output fallback for empty rows
If all 2048 sparse entries are invalid for a given `(t,h)`:
- leave output vector as all zeros
- set `lse = -inf`

This is mathematically correct and avoids NaN propagation.

#### Safety Rule 4: Conservative fallback kernel
If any of the following conditions fail, dispatch a conservative but correct fused scalar/SIMT kernel:
- tensors are not CUDA
- dtypes mismatch expected types
- tensors are not contiguous
- architecture-specific vectorization assumptions are unsafe

This fallback should still use the same online recurrence and should never allocate large temporary tensors.

#### Safety Rule 5: Base-2 LSE enforcement
The final formula must be explicitly:
- `lse = m * 1.4426950408889634f + __log2f(l)`
not `logf`, not `log2(exp(m)*l)` in a numerically unstable form.

### 3.2 Deterministic post-design validation checklist for executor
Before finalizing code, the executor should validate these invariants:
1. Every BF16-to-float conversion uses `__bfloat162float`, `__bfloat1622float2`, or equivalent explicit intrinsic.
2. Every float-to-BF16 store uses `__float2bfloat16` or `__floats2bfloat162_rn`.
3. No path writes `O_tmp`, `m_tmp`, or `l_tmp`-style workspaces.
4. Output and `lse` shapes exactly match the contract.
5. `PYBIND11_MODULE` exists and the entry point name is exactly `dsa_forward`.

### 3.3 Final repair/fallback mechanism
If during implementation register pressure makes the fully fused small-path unstable:
- keep the fused algorithm,
- but reduce register blocking in output accumulation,
- or compute output in two sequential 256-d halves inside the same kernel.

This preserves correctness while avoiding the invalid fallback of reintroducing massive temporary tensors.

## Phase 4: Implementation Details

This phase specifies exact coding actions and allowed libraries.

### 4.1 Allowed libraries and headers
Use only standard CUDA/PyTorch extension headers already available:
- `#include <torch/extension.h>`
- `#include <cuda_bf16.h>`
- `#include <cuda_runtime.h>`
- optionally `#include <ATen/cuda/CUDAContext.h>` if stream access is needed
- optionally `#include <c10/cuda/CUDAGuard.h>` for device guard

Do **not** depend on obscure packages.
Do **not** require external autotuning frameworks.

### 4.2 Data layout assumptions
Treat input buffers as contiguous flattened arrays:
- `q_nope`: contiguous over `[T][16][512]`
- `q_pe`: contiguous over `[T][16][64]`
- `ckv_cache`: contiguous over `[num_pages][64][512]`
- `kpe_cache`: contiguous over `[num_pages][64][64]`

Because `sparse_indices` encodes `(page_idx * 64 + offset)`, the flattened `idx` can directly index the second dimension of cache tensors collapsed into `[num_pages*64, dim]` semantics.

### 4.3 Recommended kernel decomposition
Implement three device-level pieces:

1. `warp_reduce_sum(float x)`
   - Use `__shfl_down_sync`

2. `process_sparse_tile(...)`
   - Helper that loads one 32-entry sparse tile to shared memory and updates online state for one warp/job

3. `dsa_fused_small_kernel(...)`
   - Persistent or grid-stride small-workload kernel

4. `dsa_fused_large_kernel(...)`
   - Standard fused kernel for large workloads

5. `dsa_forward(...)`
   - Dispatcher and tensor validation

### 4.4 Dispatcher logic
Use a deterministic dispatch rule such as:
- `if (num_tokens <= tau_small) -> small persistent fused path`
- `else -> large fused path`

Start with `tau_small = 192` or `256`.
The child should prefer robustness over overfitting; a single threshold is acceptable.

### 4.5 Vectorized access plan
For `ckv_cache` and `kpe_cache` staging:
- use 128-bit loads where safe via `float4`/`int4` reinterpretation
- load shared-memory rows cooperatively
- maintain alignment assumptions by checking contiguity and using naturally aligned base pointers

For query loads:
- use paired BF16 loads as `__nv_bfloat162` where convenient
- convert immediately to FP32 fragments

### 4.6 Register blocking plan
Store per-lane output fragments in registers:
- baseline: `float2 acc[8]` for 16 BF16 values per lane across 512-d output mapping
- if resources permit, expand the blocking
- if register spill appears likely, keep the baseline and prioritize occupancy

### 4.7 Shared-memory plan
Shared memory should hold only the current sparse tile:
- `int idx_shared[32]`
- BF16 staged keys/values for 32 rows

Do not allocate multi-stage global workspaces.

### 4.8 Error checking and robustness in C++ entry point
Inside `dsa_forward`, explicitly check:
- CUDA device tensors
- BF16 dtype for `q_nope/q_pe/ckv_cache/kpe_cache`
- int32 dtype for `sparse_indices`
- correct ranks and fixed head/dim extents
- contiguous layout

If any check fails, use `TORCH_CHECK` with a clear message.

### 4.9 Numerical detail checklist
- Use `__expf` for exponentials in the recurrence.
- Compute `lse` with `__log2f`.
- If `l <= 0`, store zero output and `-INFINITY` lse.
- Never cast BF16 directly to float with C-style casts.

## Why this plan should outperform the parent

### Expected performance improvement
The parent already performs well for large workloads. Its weakest point is the low-workload split-K path, which:
1. launches two kernels,
2. writes large FP32 temporaries to HBM,
3. reads them back for reduction,
4. still underfills the GPU.

The child should improve by:
- replacing split-K with a fused online-softmax kernel,
- eliminating `O_tmp`, `m_tmp`, `l_tmp`,
- reducing launch overhead,
- reducing HBM traffic,
- keeping exact recurrence semantics.

### Advantages
- Strong correctness story.
- Deterministic and robust.
- Directly addresses the bottleneck identified in the parent summary.
- Uses the parent’s proven strengths instead of discarding them.

### Disadvantages
- Less ambitious than a full tensor-core sparse MMA redesign.
- May not dramatically improve already-saturated large-T workloads.
- Requires careful register-pressure management.
