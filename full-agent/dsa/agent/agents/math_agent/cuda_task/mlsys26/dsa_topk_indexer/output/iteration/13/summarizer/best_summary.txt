**3. Strategic Analysis (The "So What?"):**
*   **Root Cause:** **Plan was excellent; execution was correct but suboptimal.** The fusion concept was well-conceived and the partition invariant (each batch handled by exactly one path) was preserved. However, the result (8.16) is *lower* than the pure fast-path sibling (8.97) despite adding a supposedly-orthogonal speedup. Likely cause: benchmark variance (the reference baseline shifted again, 2.79ms → 2.91ms), plus the uint4 path's marginal gain on long sequences may be drowned out because the dominant cost on the long-path is now the CUB segmented sort, not the scoring kernel. The two optimizations are orthogonal in *theory* but the long-path scoring is no longer on the critical path.
*   **Key Insight:** **Once a structural shortcut (fast-path) removes the bulk of work, further micro-optimizations on the bypassed path produce near-zero observable gain** — Amdahl's Law in action. The scoring-kernel uint4 work has effectively reached the point of diminishing returns. The next dominant cost is unambiguously the CUB device-wide segmented sort on the long batches.
*   **Identified Risk:** Continued investment in scoring-kernel micro-optimizations (cp.async pipelining, more vectorization, tensor cores) will likely yield <5% total improvement because long-sequence batches are now a small fraction of total runtime. Worse, fusing more complex changes risks correctness regressions for diminishing benefit. The score gap to sibling 8.97 also signals possible noise sensitivity at this latency floor (~0.045ms).

**4. Actionable Guidance (The "What's Next?"):**
*   **Recommend Exploration (HIGH PRIORITY):** Attack the **CUB device-wide segmented radix sort** on the long path. Replace it with either (a) a per-batch **block-level `cub::BlockRadixSort`** in shared memory when `L ≤ ~8192`, or (b) a **two-stage bucketed top-K**: first a per-`(batch, page)` block emits local top-32 candidates (reducing input from `N` to `~max_num_pages*32`), then a smaller CUB sort. Expected gain: 20–40% on long-sequence workloads.
*   **Recommend Exploration:** Add a **medium-sequence tier** (`2048 < L ≤ 4096`): use a single-kernel block-level radix-select fused with scoring, eliminating the device-wide CUB call entirely for that range.
*   **Recommend Stripping:** Consider reverting the uint4 K-load to scalar `uint32_t` (matching `2fb50fd2`) if it simplifies code without measurable loss — sibling `2fb50fd2` proves the fast-path alone reaches 8.97.
*   **Warn:** Do NOT add further micro-optimizations to `compute_scores_kernel` until the CUB sort bottleneck is addressed — Amdahl's wall is near. Preserve the strict partition invariant (each batch written by exactly one path) as a non-negotiable correctness contract in any future fusion.
*   **Recommend Fusion:** The winning sibling `2fb50fd2` (score 8.97) and this child (8.16) are functionally near-equivalent; treat the higher-scoring `2fb50fd2` as the canonical baseline going forward to avoid carrying complexity (uint4 lane-masked load) that does not pay rent.
