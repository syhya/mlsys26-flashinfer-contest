# Task
You will be given with a JSON task definition describing a GPU kernel operation. Your goal is to implement a **highly optimized CUDA kernel** that significantly outperforms the provided PyTorch reference implementation while maintaining strict numerical correctness.

The definition includes:

2. **Axes**: dimension parameters (some constant, some variable like batch_size)
3. **Constraints**: relationships between dimensions
4. **Reference implementation**: a correct but unoptimized Python/PyTorch `run()` function

# Build Environment Constraints (CRITICAL)
- The build targets **SM90+ (Hopper/Blackwell)** architecture with CUDA 12+.
- `float8_e4m3fn` tensors are passed as `torch::Tensor` with dtype `torch::kFloat8_e4m3fn`. In CUDA device code, access the raw bytes as `uint8_t` or `__nv_fp8_e4m3` and convert to float explicitly.
- Always `#include <cuda_fp8.h>` for FP8 intrinsics if needed.

# FP8 KV Cache Format (deep_gemm layout -- CRITICAL)
The `k_index_cache_fp8` tensor uses the **deep_gemm packing format**:
- Tensor shape: `[num_pages, page_size=64, 1, 132]` stored as `int8` (interpret as `uint8`)
- **Per-page flat layout** (viewing as `[num_pages, page_size * 132]`):
    - First `page_size * 128` bytes: FP8 (e4m3fn) key data, shape `[page_size, 128]`
    - Last `page_size * 4` bytes: float32 per-token scales, shape `[page_size, 1]`
- To dequantize token `t` in page `p`: `K_float[t, d] = fp8_to_float(fp8_data[t, d]) * scale[t]`

# Correctness Requirements
1. Output shape: `[batch_size, 2048]` with dtype `int32`.
2. The kernel computes per-batch: `scores = sum_heads(relu(q @ K.T) * weights)` then selects top-2048 indices.
3. Output values are **global token indices** encoded as `page_idx * 64 + offset_in_page`, where `page_idx` comes from `block_table`.
4. Positions beyond `seq_lens[b]` must NOT contribute to scores. Indices beyond the valid top-K count must be `-1`.
5. Top-K index **ordering does not need to match** the reference exactly, but the **set of selected indices** must be correct (i.e., correspond to the same top-2048 scores). Ties may be broken arbitrarily.
6. All variable axes (like `batch_size`, `num_pages`, `max_num_pages`) must work for any valid value.
7. Constraint: `topk (2048) <= max_num_pages * page_size (64)`.

# Core Optimization Strategy
Your solution must demonstrate a **substantial architectural or performance improvement** over a naive implementation. Superficial changes (e.g., renaming variables, simple loop reordering) are forbidden. You must innovate in at least **one** of the following dimensions:

## 1. Scoring Stage Optimization
The scoring computation `q @ K.T` is the dominant cost. This is essentially a batched matrix multiplication between FP8 queries `[64, 128]` and paged KV cache `[seq_len, 128]` with per-token dequantization.
- **FP8 Tensor Core utilization**: Leverage Hopper/Blackwell FP8 MMA instructions for the matmul.
- **Paged KV cache tiling**: Design tile strategies that respect page boundaries (64 tokens per page) to avoid scattered memory access.
- **Multi-path dispatch**: Different workload sizes (few pages vs. many pages) may benefit from different kernel strategies.

## 2. TopK Selection Strategy
Selecting top-2048 from potentially large score arrays requires an efficient selection algorithm.
- Consider: **radix-based selection**, **histogram/counting**, **bitonic sort**, **partial sort**, or **warp-cooperative approaches**.
- The choice may depend on the score array length relative to K=2048.

## 3. Memory & Pipeline Optimization
- **Asynchronous Data Movement (`cp.async` / TMA):** On SM90+ architectures, utilize asynchronous memory operations to overlap data movement and computation.
- **Vectorized Access:** Use 128-bit vectorized loads for FP8 data to maximize memory bandwidth.
- **Shared Memory Tiling:** Tile KV pages into shared memory for efficient reuse across heads.

## 4. Kernel Fusion
- Fusing the score computation, ReLU activation, weighted head reduction, and top-K selection into fewer kernel launches reduces HBM round-trips and launch overhead.

## 5. Library Usage
You are encouraged to call third-party CUDA libraries directly from `kernel.cu` when they provide a performance advantage. Available libraries in the build environment include:
- **CUB** (`<cub/cub.cuh>`): block-level and warp-level primitives (e.g., `cub::BlockRadixSort`, `cub::WarpReduce`, `cub::BlockReduce`).
- **CUTLASS / CuTe** (`<cute/tensor.hpp>` etc.): tile-level abstractions, FP8 MMA wrappers, and layout utilities for tensor-core matmul.
- **Standard CUDA math** (`<cuda_fp8.h>`, `<cuda_bf16.h>`): FP8/BF16 arithmetic helpers.
